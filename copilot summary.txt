1) PROJECT OVERVIEW
-------------------
CCTRS is a full-stack web platform that tracks environmentally positive user activities
(e.g., tree-related actions, sustainability actions), validates evidence (proof),
and rewards users with points and badges. It includes:
- User-facing activity submission and progress views
- Admin moderation, question/config management, and approvals
- Analytics and leaderboard visibility
- Authentication and protected role-based dashboards

Primary goals:
- Encourage eco-friendly behavior through gamification
- Ensure trust via proof verification and moderation flows
- Provide transparent ranking and monthly/overall progress
- Support future growth with modular backend/frontend structure

2) SYSTEM ARCHITECTURE
----------------------
Architecture style:
- Client-server web architecture
- Frontend SPA (React) communicates with REST APIs
- Backend (Spring Boot) handles business logic, auth, persistence
- Relational database with SQL schema + migration scripts

High-level components:
1. React Frontend (`frontend/src`)
   - Route-based page modules for public/user/admin use cases
   - API wrapper layer (`src/api`) for HTTP communication
   - Context-driven auth state management (`src/context/AuthContext.jsx`)
2. Spring Backend (`backend/cctrs-backend/src/main/java`)
   - Layered design: Controllers -> Services -> Repositories -> Entities
   - Security module for authentication/authorization
   - Activity/proof/approval/points domain logic
3. Database (`database/`, backend resources, running DB files under `data/`)
   - Core entities for users, activities, proofs, points, badges, leaderboard
   - Migration scripts for evolving schemas
4. Documentation (`docs/`)
   - Detailed API reference and architecture documentation
   - ER diagram and implementation planning docs

Operational flow:
- User authenticates -> submits activity/proof -> system stores as pending ->
  admin verifies/rejects -> points awarded/revoked -> badges/leaderboard updated ->
  dashboards and analytics reflect status.

5) AUTHENTICATION FLOW
----------------------
Observed/authenticated UX flow (from frontend structure + standard Spring/React patterns):

1. User accesses public/auth screens
   - Home/help/FAQ/journey and auth endpoints
2. Login or registration request
   - Frontend uses `src/api/auth.js`
   - Backend validates credentials, issues auth token/session
3. Auth state persistence
   - `AuthContext.jsx` stores user identity + role + token state
4. Route protection
   - `ProtectedLayout.jsx` blocks unauthenticated routes
   - Role-aware layout/router directs to user/admin dashboards
5. Password recovery
   - `ForgotPassword.jsx` handles reset request and follow-up flow
6. Sessionized API requests
   - `axios.js` attaches token headers/interceptors
7. Unauthorized handling
   - Expired/invalid token -> logout/redirect to login/public page

Role model:
- At least two roles implied: USER and ADMIN
- Admin-only sections exposed through admin layout and guarded routes

6) ACTIVITY SUBMISSION WORKFLOW
-------------------------------
Primary user flow:
1. User navigates to submit activity page (`SubmitActivity.jsx`)
2. User selects activity type and fills required details/questions
3. User starts proof flow (`StartProof.jsx`) if evidence is required
4. Frontend sends submission payload to backend activity endpoints
5. Backend validates:
   - Required fields
   - Activity-specific constraints
   - (Potential abuse flags / duplicate checks from migration intent)
6. Submission stored with initial status (likely PENDING)
7. User sees status in `MyActivities.jsx`
8. Upon moderation decision:
   - APPROVED -> points granted
   - REJECTED -> rejection reason displayed
9. Monthly/overall metrics update for user dashboard

Likely additional details:
- Question-driven submissions managed by admin question config
- Search and filtering available via `SearchActivities.jsx`
- Proof attachment metadata associated with submission records

7) PROOF VERIFICATION SYSTEM
----------------------------
Proof verification appears as a dedicated moderation domain.

Key parts:
- User proof intake (`StartProof.jsx`) and backend proof storage metadata
- Verification queue exposed to admin interfaces
- Moderation outcomes:
  - Verified/Approved
  - Rejected with reason (migration indicates rejection reason support)
- Abuse safeguards:
  - Tree abuse flags migration suggests anti-fraud/quality controls
- Audit/state tracking:
  - Status fields and possibly timestamps/reviewer attribution

Probable state transitions:
DRAFT/NEW -> PENDING_REVIEW -> APPROVED | REJECTED | FLAGGED

System impact:
- Approved proofs unlock points and progression
- Rejections feed user transparency and resubmission behavior
- Flagged patterns enable stronger review processes
